Promises

an elegant way to handle asynchronous operations

  • Historically, many Javascript apis used callback functions to handle asynchronous responses
    • These worked ok, but have some issues
    • No standardized way of doing things (how many callback functions, what are the arguments?)
    • Results in pyramid code - deeply nested functions
    • No good way to do error handling (each parent function has to handle errors of its descendants)
  • Async apis in Angular return a promise object, which represents the result of an action that may or may not be finished yet.
  • The standard way to resolve a promise is with .then(successCallback, errorCallback, notifyCallback)
  • .then() itself returns a promise, which is resolved or rejected by the return value of these callbacks
    • This is the key to chaining promises without deep nesting
    • Built in support for exception handling, such that throwing an exception will reject the promise
    • Can also explicitly reject a promise using return $q.reject(reason);
    • The callback can return another promise, when nested chaining is necessary
    • Returning any other value will be considered successful resolution
  • New promises are created using Angular's $q service
    • The implementation was influenced by Kris Kowal's Q
